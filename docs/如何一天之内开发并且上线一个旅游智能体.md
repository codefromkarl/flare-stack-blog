# 如何一天之内开发并且上线一个旅游智能体

> **全栈 AI 应用从零到上线的 24 小时实录**

---

## 产品简介

**旅图 TravelMap**（https://travel.codefromkarl.xyz）是一个 AI 驱动的智能旅行规划助手。用户输入目的地和偏好，系统自动生成行程路线、景点推荐、住宿建议、天气预报和预算规划。

这个项目从第一个代码提交到正式上线，大约经历了 24 小时。本文以 git 提交记录为主线，按时间线复盘整个开发过程，每个阶段标注了起止时间。

---

## 技术栈概览

- **Agent 框架**：[pi](https://github.com/earendil-works/pi) — AI Agent 编排框架，负责调度大模型推理与工具调用
- **前端**：Web Components（Lit）+ Leaflet.js 地图
- **部署**：Cloudflare Pages + Functions（边缘计算平台，全球加速）
- **测试**：Vitest（单元/集成测试）+ Playwright（端到端测试），共 542 个自动化测试
- **代码质量**：Biome（代码风格）+ TypeScript strict（类型安全）

**核心能力清单**：
- 交互式地图（标准/卫星/地形图层切换）
- 多模型 AI（OpenAI / Anthropic / Google / DeepSeek / OpenRouter）
- 智能行程规划（每日行程、景点、住宿、城际交通）
- 实时天气与出行风险评估
- 多人群适配（成人/老人/儿童/孕妇）
- 行动链接（一键预订、比价、搜索跳转）
- 多格式导出（Markdown / PDF）
- 国际化（中文 / English / 日本語）

214 行的测试文件定义了后续所有服务测试的写法范式。在代码还未被大量依赖之前锁定接口契约，降低了后续重构的成本。

### 16:14 HTTP 客户端统一

```bash

```

所有外部 API 调用需要一个统一的 HTTP 客户端，具备重试、超时控制能力。后续 xhs（小红书数据源）、elevation（海拔查询）、weather（天气）等服务均基于此客户端构建，保证了错误处理策略的一致性。

### 16:39 服务层重构 + 架构守卫

```bash
refactor(services): xhs/elevation 接入统一 http-client，添加架构守卫与规范文档
```

将小红书数据抓取和海拔查询服务接入统一 HTTP 客户端。同时引入了**架构守卫（Architecture Guard）**——一种在 CI 中自动检测架构违规的机制，确保代码演进过程中不偏离既定架构规范。

### 17:09 编排契约测试

```bash
test(integration): 编排契约测试 — 验证景点→天气→坐标并行搜索链路
```

**集成测试**验证 Agent 编排器能否正确并行调度多个搜索服务（景点 → 天气 → 坐标）。这种测试验证的是真实数据流而非单一模块行为，价值高于单元测试。

> 契约测试（Contract Test）是此项目的核心测试策略：不 Mock 内部实现，而是验证"输入 X 时输出是否符合预期形状"。测试对重构免疫，是大规模重构的前提保障。

### 17:24 - 17:34 Provider Adapter 模式

### P4 - 导出与国际化

```

```

- **导出**：Markdown 和 PDF 两种格式，使用 SheetJS 生成可下载文件
- **国际化**：中文 / English / 日本語 三语界面，文案集中管理

### P5 - 高级规划能力

```
archive 05-18-p5-companion         # 多人群适配
archive 05-18-p5-model-handoff     # 模型切换
archive 05-18-p5-multi-city        # 多城市规划
archive 05-18-p5-session-persist   # 会话持久化
```

规划引擎的核心差异点：

- **多人群适配**：成人和儿童的推荐差异显著（景点类型、用餐时间、步行强度），系统根据人群组合动态调整搜索策略
- **模型切换（Model Handoff）**：预搜索用低成本模型（如 DeepSeek），推理规划用高能力模型（如 Claude），在 token 消耗与输出质量之间取得平衡
- **多城市规划**：支持复合行程（如"东京 5 天 + 大阪 3 天"），城际交通自动衔接
- **会话持久化**：页面刷新后不丢失对话历史

### P6 - Token 优化与后处理

```
archive 05-18-p6-token-opt          # Token 优化
archive 05-18-p6-search-pre-orch    # 预搜索编排
archive 05-18-p6-post-process       # 后处理管线
archive 05-18-p6-split-prompt       # Prompt 拆分
archive 05-18-p6-on-demand-schema   # 按需 Schema
```

Token 优化是项目最具工程价值的模块之一，围绕一个核心目标：**减少大模型调用次数和上下文长度**。

- **预搜索编排（Pre-search Orchestration）**：在 LLM 调用前，系统并行执行天气、景点、餐厅等搜索服务，将结果直接注入 Prompt。LLM 无需通过 Tool Call 逐个查询，节省约 80% 的来回调用
- **Prompt 拆分**：将超长 System Prompt 拆分为多个独立片段，按阶段注入（偏好挖掘 → 行程生成 → 微调），降低单次调用的上下文窗口压力
- **按需 Schema（On-demand Schema）**：初始对话使用精简 Schema，仅在需要时加载完整 Schema 定义
- **后处理管线（Post-process Pipeline）**：LLM 输出原始行程后，系统自动执行预算计算、行动链接注入、格式校验，节省 2-4 次额外 LLM 调用

### P7 - 消息压缩

```
archive 05-18-p7-compact-search     # 紧凑搜索结果
archive 05-18-p7-message-compress   # 历史消息压缩
```

长对话面临的上下文窗口溢出问题。解决方案：当消息数超过阈值（8 条）时，自动压缩旧轮次为摘要，保留最近 1 轮完整对话。

### P8 - 模型路由与差分 Steering

```
archive 05-18-p8-model-router       # 模型路由
archive 05-18-p8-steering-diff      # 差分微调
```

- **模型路由**：根据任务特性自动选择 LLM 供应商——简单分类用 DeepSeek（低成本），推理规划用 Claude（高智商），创意写作用 GPT-4（均衡）
- **差分 Steering（Diff-based Steering）**：用户要求"把第二天改松一点"时，系统只修改差异部分而非重新生成整个行程，显著降低 token 消耗和生成延迟

## 关键经验

1. **测试契约先行**：核心价值来自契约测试（验证接口形状而非内部实现），这让大规模重构成为可能。没有这套测试体系，570 行重构净减的结果不可能安全落地。

2. **Token 优化是 AI 应用的性能瓶颈**：预搜索编排 + Prompt 拆分 + 消息压缩 + 差分 Steering，四层优化决定了用户体验从"不可用"到"流畅"的质变。

3. **质量门禁设底线**：75/70/65 的覆盖率阈值不是指标考核，而是安全底线——确保重构不会无声地破坏功能。

4. **模块边界尽早确立**：4700 行单文件的教训是：模块拆分应在功能完整后立即执行，拖延越久重构成本越高。

5. **部署与 CI 解耦**：独立 deploy workflow 是一个小改动，但在紧急修复场景下价值巨大——CI 失败不会阻塞回滚部署。

6. **上线是迭代起点**：上线后一小时内的城际交通查询、地图 POI 反查、E2E 测试重构说明：产品的真正价值在于持续响应真实使用场景。
